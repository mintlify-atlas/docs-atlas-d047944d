---
title: "VR Support via Nexus VR Core Integration"
description: "LocalAudio Subtitles automatically switches to a 3D in-world subtitle display when running in VR, using the Nexus VR Core ScreenGui3D component."
---

LocalAudio Subtitles includes automatic VR support. When the module detects that the player is in a VR session, it replaces the standard `ScreenGui` with a Nexus VR Core `ScreenGui3D` — a three-dimensional in-world panel that stays readable in headset. No code changes are needed in your subtitle definitions; VR handling is entirely transparent.

## How it works

When a subtitle window is initialised, the module checks `UserInputService.VREnabled`. If VR is active, `LoadNexusVRCore()` is called, which:

1. Requires the `ScreenGui3D` class from `NexusVRCore` in `ReplicatedStorage`.
2. Creates a new `ScreenGui3D` instance and configures it.
3. Reparents the subtitle row frame into the new VR container.
4. Destroys the original flat `ScreenGui`.

The VR screen is configured with the following properties:

| Property | Value |
|---|---|
| `Name` | `LocalAudioSubtitlesVR` |
| `DisplayOrder` | `100` |
| `ResetOnSpawn` | `false` |
| `CanvasSize` | `Vector2.new(1200, 800)` |
| `PointingEnabled` | `false` |

`PointingEnabled = false` prevents the VR laser pointer from interacting with the subtitle panel, keeping it as a passive display surface.

<Info>
  If the player is not in VR, the module uses a standard `ScreenGui` exactly as normal. No special configuration is needed for non-VR players — everything is handled automatically.
</Info>

## Requirements

<Steps>
  <Step title="Add Nexus VR Core to ReplicatedStorage">
    Place the Nexus VR Core module in `ReplicatedStorage` and name it exactly `NexusVRCore`. The module listens for this child at runtime using `WaitForChild`, so it can be loaded at any point — it does not need to exist before the subtitle window is created.
  </Step>
  <Step title="Add the Wally dependency">
    If your project uses [Wally](https://wally.run), add Nexus VR Core to your `wally.toml`:

    ```toml wally.toml
    [dependencies]
    NexusVRCore = "thenexusavenger/nexus-vr-core@2.2.2"
    ```

    Then run `wally install` to fetch the package.
  </Step>
</Steps>

<Warning>
  The VR integration path (`LoadNexusVRCore`) only runs when the subtitle container is a `ScreenGui`. If you have already replaced it with a custom container, the VR swap will not occur. Ensure your subtitle window is initialised with a standard `ScreenGui` when VR support is required.
</Warning>

## Subtitle Engine integration in VR

LocalAudio Subtitles also supports the [Subtitle Engine](https://devforum.roblox.com/t/subtitle-engine/3781802) library for typewriter-style subtitle rendering. When a subtitle entry is shown via the Subtitle Engine (`ShowSubtitleEngine`), the engine uses the `Self` profile and applies a per-character type delay based on the entry's duration:

- If the entry has a speaker, the speaker name is displayed in bold with no type delay, while the message body types in over half the subtitle's duration.
- If there is no speaker, the full message types in with a fixed delay of `0.03` seconds per character.

This behavior applies the same way in VR — the Subtitle Engine writes to the `Self` profile regardless of whether the display is a flat `ScreenGui` or a `ScreenGui3D`.

```luau
-- Speaker present: name appears instantly, message types over Duration * 0.5 seconds
SubtitleEngine.editProfile("Self", { extraWaitTime = self.Duration * 0.5 })
SubtitleEngine.localSubtitle("Self",
    '<typeDelay = 0><b>' .. self.SpeakerName .. '</b>: </typeDelay>' ..
    '<typeDelay = ' .. typeDelayPerChar .. '>' .. self.Message .. '</typeDelay>'
)

-- No speaker: full message types with fixed 0.03s per character delay
SubtitleEngine.editProfile("Self", { extraWaitTime = self.Duration })
SubtitleEngine.localSubtitle("Self", '<typeDelay = 0.03>' .. self.Message .. '</typeDelay>')
```
